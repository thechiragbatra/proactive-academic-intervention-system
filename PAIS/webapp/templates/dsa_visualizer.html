01

Priority Queue · max-heap

push: O(log n) · top: O(1)

Concept

A max-heap keeps the highest-risk student at the root. The mentor's dashboard needs the top 10 worst cases, and updates happen often (edits, what-if simulations). Re-sorting 5,000 entries every change is O(n log n); a heap update is O(log n) and reading the top is O(1).

Python's heapq is a min-heap, so we store -score to flip it. A monotonic counter breaks ties.

Core code

import heapq

class RiskHeap:
    def __init__(self):
        self._heap = []
        self._counter = 0

    def push(self, sid, score, meta):
        # negate score → max-heap behaviour
        heapq.heappush(self._heap,
                       (-score, self._counter, sid, meta))
        self._counter += 1

    def peek_top(self, k=1):
        return heapq.nsmallest(k, self._heap)

02

Sliding Window · attendance anomaly

O(d) per student · d = days

Concept

Aggregate attendance hides short collapses. A student at 85% overall can have a one-week blackout that signals trouble. We slide a 7-day window across each student's daily log and flag windows where the mean drops below 50% or variance is too high.

Trick: keep a running sum. Adding the new day and subtracting the leaving day is O(1) per step, so the whole scan is O(d) instead of O(d × window).

Core code

def scan(days, attended, w=7):
    # running sum trick — O(d) total
    s = sum(attended[:w])
    for i in range(len(attended) - w + 1):
        mean = s / w
        if mean < 0.5:
            yield (i, i + w - 1, mean)
        if i + w < len(attended):
            s += attended[i + w]      # enters
            s -= attended[i]          # leaves

03

Hash Map · O(1) student lookup

avg O(1) per get/put

Concept

Each click on a student profile builds a view by joining academic data, daily logs, risk score, and recommendations. With a DataFrame scan that's O(n) per click. A hash map (Python's dict under the hood) gives us O(1) average lookup.

We wrap the dict in a thin class so call sites stay clean and the storage can swap out (Redis, sqlite) without touching them.

Core code

class StudentHashIndex:
    def __init__(self):
        self._table = {}    # Python dict

    def put(self, sid, profile):
        self._table[sid] = profile

    def get(self, sid):
        return self._table.get(sid)

    def __len__(self):
        return len(self._table)

04

Bipartite Graph · BFS for engagement

BFS: O(V + E)

Concept

Engagement isn't one-dimensional. A student might attend classes but never touch course materials. We model the access pattern as a bipartite graph — students on one side, course resources on the other, edges weighted by access count.

A BFS from the core resources gives us the set of "reachable" students. Anyone with fewer than 2 distinct resource ties is flagged as isolated.

Core code

from collections import defaultdict, deque

class ResourceGraph:
    def __init__(self):
        self.adj = defaultdict(dict)

    def add_edge(self, sid, res, w=1):
        self.adj[f"S:{sid}"][f"R:{res}"] = w
        self.adj[f"R:{res}"][f"S:{sid}"] = w

    def bfs(self, start):
        seen, q = {start}, deque([start])
        while q:
            n = q.popleft()
            for nb in self.adj[n]:
                if nb not in seen:
                    seen.add(nb); q.append(nb)
        return seen

05

Greedy · grade roadmap

O(g) · g = grade tiers

Concept

Telling a student "you are at risk" is useless. Telling them "you need 67% on the final to scrape an A" is actionable. For each grade target, we compute the minimum percentage on the remaining evaluations.

It's greedy because we walk grade targets from highest cutoff downwards and commit to a uniform required percentage at each step — no backtracking. Targets where the percentage exceeds 100 are marked infeasible.

Core code

def recommend_for_grade(target, cutoff, earned,
                        remaining_weight):
    gap = cutoff - earned
    if gap <= 0:
        return {"feasible": True,
                "required_pct": 0,
                "note": "already locked"}

    pct = gap / remaining_weight
    return {"feasible": pct <= 100,
            "required_pct": pct,
            "note": build_note(pct)}

06

Stable Sort · improvement gradient

O(n log n) · mergesort

Concept

The Analytics page surfaces top improvers (students whose midterm jumped vs their pre-midterm average) and fastest decliners. The gradient is midterm − early_avg. Positive means improving, negative means declining.

We pass kind="mergesort" explicitly because mergesort is guaranteed stable. Two students with identical gradients keep their original order across runs.

Core code

def rank_by_gradient(df):
    df = df.copy()
    df["gradient"] = (
        df["Midterm_Score"]
        - df["early_academic_avg"]
    )
    # mergesort → guaranteed stable
    return df.sort_values(
        "gradient",
        ascending=False,
        kind="mergesort",
    )
